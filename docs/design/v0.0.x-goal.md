# v0.0.x Goal

The v0.0.x goal is to centralize target AI-native configuration definitions and materialize them into target AI project paths.

This phase focuses on centralized definitions, not exact reproducibility. Individual Git sources can be pinned to an exact revision, but v0.0.x offers no whole-run reproducibility guarantee such as a lockfile.

## Target AI

v0.0.x supports Claude and Codex.

Supported materialization targets:

```text
claude + skill -> .claude/skills/<name>/
claude + agent -> .claude/agents/<name>.md
codex  + skill -> .agents/skills/<name>/
codex  + agent -> .codex/agents/<name>.toml
```

Enozunu materializes each source into the selecting target's native path without converting the artifact between target formats. The responsibility boundary is recorded in [the Claude and Codex materialization ADR](adr/20260711T184657Z_materialize-claude-and-codex-without-semantic-conversion.md).

Unsupported in v0.0.x:

```text
consumer targets other than claude and codex
AGENTS.md
.codex/config.toml
Codex rules, MCP, hooks, and plugins
Claude and Codex agent format conversion
target-native semantic validation
```

## Source of Truth

The primary source of truth is:

```text
enozunu.kdl
```

Generated output is not source of truth. For v0.0.x, `.claude/`, `.agents/`, and `.codex/` may be generated output.

The machine-generated derived record is:

```text
.enozunu/provenance.json
```

`provenance.json` records what was materialized.
It is not a lockfile.
It is not used as a resolution input in v0.0.x.

## Supported Source Reference

v0.0.x supports the normalized source reference shape:

```kdl
skill "git-kura" {
  git {
    url "https://github.com/tooppoo/reportage"
    branch "main"
    path ".claude/skills/git-kura"
  }
}
```

Each source declares exactly one source reference block. A `git` block uses:

```text
url + exactly one of (branch, revision) + path
```

`branch` follows the current head of a branch; `revision` pins one exact commit as a canonical full SHA-1 commit id. Declaring both selectors, or neither, is invalid. The selector contract is recorded in [the Git exact revision selector ADR](adr/20260712T155345Z_git-exact-revision-selector.md).

`local` and `gist` blocks are also supported. See [the manifest format guide](../guide/manifest.md#source-reference-blocks) for the full source reference contract. GitHub tree/blob URL shorthand is not supported.

## Supported Artifacts

A Skill source is a directory containing `SKILL.md`.
The whole directory is copied to the target path, including supporting files.
Claude and Codex use the same Skill artifact shape.

An agent source is a file. It is copied to the selecting target's native agent path: `.claude/agents/<name>.md` for Claude and `.codex/agents/<name>.toml` for Codex. A Claude agent is a Markdown file and a Codex custom agent is a TOML file; Enozunu copies the file verbatim and does not convert between the two.

v0.0.x does not validate which target AI a source was originally created for, and it does not validate the target-native agent format. It validates the artifact shape needed for materialization.

## Manifest Shape

```kdl
enozunu config-version=1 {
  provider {
    skills {
      skill "git-kura" {
        git {
          url "https://github.com/tooppoo/reportage"
          branch "main"
          path ".claude/skills/git-kura"
        }
      }
    }

    agents {
      agent "shell-script-reviewer-claude" {
        git {
          url "https://github.com/tooppoo/installerer"
          branch "main"
          path ".claude/agents/shell-script-reviewer.md"
        }
      }

      agent "shell-script-reviewer-codex" {
        git {
          url "https://github.com/tooppoo/installerer"
          branch "main"
          path ".codex/agents/shell-script-reviewer.toml"
        }
      }
    }
  }

  consumer {
    claude {
      use-skills "git-kura"
      use-agents "shell-script-reviewer-claude"
    }

    codex {
      use-skills "git-kura"
      use-agents "shell-script-reviewer-codex"
    }
  }
}
```

## Generated Output Policy

Skill directory materialization uses replace semantics, not merge semantics.

When a Skill source removes a supporting file, that file should also disappear from the materialized target after regeneration.

Enozunu does not preserve, detect, or integrate manual edits inside generated output.

## Provenance

At materialization time, every Git selector is resolved to a commit hash: a branch selector resolves to the current branch head, and a revision selector materializes exactly the pinned commit.

The resolved revision is written to `.enozunu/provenance.json` with enough information to inspect the previous materialization result:

- source name
- source URL
- declared selector (branch or revision, with its value)
- resolved revision
- source path
- target AI (`claude` or `codex`)
- target path

When one source is materialized to both targets, provenance records one entry per target: the source fields are identical, and the target AI and target path differ. The provenance version stays `1`.

Because branch selectors are mutable, running materialization again may produce a different result for branch-selected sources. A revision-selected source materializes the same commit on every run.

## Future Work

Reproducibility guarantees beyond per-source revision pinning are planned for v0.1.x.
Tag selectors are introduced after revision selectors.
The SHA-1-only revision contract is reconsidered when support expands to other Git hosting systems or repository formats, including GitLab and SHA-256 repositories.

Later phases may also add:

- lockfiles
- frozen materialization
- Codex `AGENTS.md`, `.codex/config.toml`, and rules, MCP, hooks, and plugins
- target digest based hand-edit detection
- generated output and manual edit reconciliation
- registry or marketplace discovery

Those are outside v0.0.x.
