# v0.1.x Goal

The v0.1.x goal is whole-run reproducibility: one unchanged manifest plus one unchanged lock file materializes the same commits on every run.

[The v0.0.x goal](v0.0.x-goal.md) deliberately excluded this. v0.0.x centralized definitions and recorded what was materialized; v0.1.x freezes what will be materialized. The decision is recorded in [the lockfile ADR](adr/20260724T021001Z_lockfile-based-reproducibility.md).

## Lockfile

The machine-generated lock file is:

```text
enozunu.lock.json
```

It lives next to the manifest, is intended to be committed, and records the resolved commit for every mutable Git selector — `branch` and `tag`. Revision-pinned Git sources, Gist sources, and local sources are not locked: their reproducible identity, or the lack of one for local sources, is already fixed by the manifest.

`.enozunu/provenance.json` is unchanged: it remains the write-only record of what the last run materialized, and it is still not a resolution input. The three per-project files split as: `enozunu.kdl` is human-authored intent, `enozunu.lock.json` is the machine-written resolution input, and `.enozunu/provenance.json` is the machine-written execution record.

## Summon Modes

`enozunu summon` is lock-first by default:

- A mutable selector with a lock entry materializes the recorded commit, verified through the exact-revision resolution path.
- A mutable selector without a lock entry resolves fresh and is added to the lock.
- A source removed from the manifest is pruned from the lock on the next run.

`enozunu summon --update` re-resolves every mutable selector and rewrites the lock. This is the only way a locked ref moves.

`enozunu summon --frozen` resolves strictly from the lock and never writes it. It fails with `lock-out-of-date` when the lock file is missing or lacks an entry for a mutable source, reporting every gap before any resolution work. This is the CI mode.

## Out of Scope

v0.1.x does not add:

- version ranges or latest selectors
- dependency solving
- registry or marketplace discovery
- strict `npm ci`-style lock-equality checking in `--frozen`
- target digest based hand-edit detection
- generated output and manual edit reconciliation

Those remain future work, unchanged from [the v0.0.x future work list](v0.0.x-goal.md#future-work).
